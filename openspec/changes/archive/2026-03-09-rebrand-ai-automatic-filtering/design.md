## Context

The feature currently called "Automatic Filtering" uses AI-generated keyword lists (produced by Claude) fetched from S3. The code/manager layer (`AutomaticFilterManager`, `AutomaticFiltersRule`, etc.) correctly reflects the feature internals, but all user-facing strings still use the old name. This is a pure copy change — no architectural or behavioral modifications.

## Goals / Non-Goals

**Goals:**
- Replace all user-visible occurrences of "Automatic Filtering" with "AI Filtering" in English
- Update non-English localizations (ar, es, fr, pt-BR, he) with the translated equivalent of "AI Filtering"
- Update the explanatory footer text on the AI Filtering screen (`help_automaticFiltering`) to clearly state that filter lists are generated by AI
- Add a periodic lightning bolt animation to the AI Filtering icon on the home screen (iOS 17+)

**Non-Goals:**
- Renaming Swift types, manager classes, CoreData entities, or internal identifiers
- Changing any filtering behavior or logic
- Updating code comments or internal documentation

## Decisions

**Keep internal naming unchanged.**
Manager names (`AutomaticFilterManager`), CoreData entities (`AutomaticFiltersRule`, `AutomaticFiltersLanguage`), and string key names (`autoFilter_title`, `autoFilter_empty`, etc.) remain as-is. Renaming them would be a large refactor with no user benefit and high regression risk. Only the localized *values* (not the keys) change.

**Update all localization files together.**
All `.strings` files (en, he, ar, es, fr, pt-BR) are updated in one pass rather than English-first with follow-up PRs. Since the translated strings use the same localization keys, a translator prompt can generate all languages at once.

**"AI Filtering" as the canonical term (not "AI Automatic Filtering").**
"Automatic" is redundant when the word "AI" is present. "AI Filtering" is shorter, fits nav bar titles, and is more consistent with how Apple labels smart/automatic features.

**Icon animation (iOS 17+ only).**
The `bolt.shield.fill` icon on `AppHomeView` uses `phaseAnimator([0, 1, 2, 1, 0], trigger:)` to create a double-spark lightning effect every 1.5 seconds. A `Task`-based sleep loop fires the trigger rather than a continuous `phaseAnimator`, giving clean control over timing and a guaranteed resting state. The animation is gated on `!isAllUnknownFilteringOn && !reduceMotion` — when either is true, a static palette-rendered icon is shown instead. On iOS 16 and below, the plain indigo icon is shown with no animation. Internal trigger state (`shieldGlintTrigger`) lives on `AppHomeView` directly (not the ViewModel) as it is pure presentation state.

## Risks / Trade-offs

- **Over-promising with "AI" label** → Mitigated: lists are genuinely AI-generated (Claude), so the claim is accurate and defensible with Apple review.
- **Translation quality for "AI Filtering"** → In most languages, "AI" is kept as-is (it's a universal acronym); only "Filtering" needs translation. Low risk.
- **Missing an occurrence** → Mitigated by grepping all `.strings` and `.swift` files for "Automatic Filter" before closing the task.
